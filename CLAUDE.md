# jobsite-v2

> **Repo:** `Angularious/jobsite-v2` ‚Äî the canonical repo. `jobenrich` was the
> boilerplate this was copied from; ignore it. **Deploy target:** Vercel project
> **jobsite-v2**. **Supabase objects:** `jobsitev2_`-prefixed. The user-facing UI
> still reads "Job Enrich" (`/`) / "Job Search" (`/jobs`) ‚Äî renaming that public
> copy is a separate product decision, intentionally not done as part of the
> runtime rename above.

A Next.js 16 app that surfaces the **people behind any job posting** ‚Äî LinkedIn, Indeed, Greenhouse, Lever, Workday, BambooHR, Gem, or a company careers page. A student/job-seeker pastes a job URL and gets people at that company worth reaching out to, plus recruiters hiring for it. They can then optionally enter their school to surface alumni, and pull contact info for any person on demand. **Powered by Orthogonal.**

## Stack

- Next.js 16.2.9 (App Router) ¬∑ React 19 ¬∑ TypeScript
- Tailwind v4 ‚Äî config lives in `app/globals.css` via `@theme`, **not** a `tailwind.config` file. Design system is **light raw brutalism** (per the `nextlevelbuilder/ui-ux-pro-max-skill` "Brutalism" spec): white paper, black ink + 3px black borders, blue underlined links, pure primary accents (red is primary; blue/yellow/green/pink alongside), **no shadows**, zero border-radius, system-ui + monospace fonts (bold 700+), instant state changes (`transition: none`). Buttons are plain bordered blocks that **invert on hover** (white/black ‚Üí black/white) ‚Äî colors live in the `.nb-btn` primitive, so they override utility bg/text classes on the element. Reusable primitives `.nb-card` / `.nb-btn` / `.nb-input` / `.nb-flat` live in `globals.css`; a fill accent (where needed) is passed via the inline `--nb` CSS custom property. Errors/alerts use hot pink to stay distinct from the red primary. **Gotcha:** `text-base` resolves to the `--color-base` **color** (white), not a font-size ‚Äî use `text-ink` for black text and an explicit size class (e.g. `text-sm`).
- `lucide-react` for icons. No test framework.

## Environment variables

- `ORTHOGONAL_API_KEY` ‚Äî auth for all external data calls (required, **server-side only**, never `NEXT_PUBLIC_`).
- `DAILY_SPEND_CAP_USD` ‚Äî hard daily ceiling across all visitors (default 40 in code). Editable in Vercel without a code change.
- `REQUEST_TOKEN_SECRET` ‚Äî HMAC secret for signing request/timing tokens (random 32+ chars).
- `NEXT_PUBLIC_APP_URL` ‚Äî public deployment origin, used for same-origin checks on API routes.
- `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` ‚Äî **optional**. When both are set (and `supabase/migrations/0001_*.sql` has been run), the spend cap + rate limiter use Supabase so they hold across all serverless instances. Unset ‚Üí the in-memory per-instance counters are used (fine for a low-traffic demo). Service-role key is server-side only ‚Äî never `NEXT_PUBLIC_`. Free tier is sufficient.

> **‚öÝ Shared Supabase project ‚Äî ONE Supabase project for ALL demo sites in `~/Projects/`:** every Orthogonal demo site in the `Projects/` folder (one level above `jobsearch/`) ‚Äî `rolodex`, `job-intel`, `jobenrich`, etc. ‚Äî **shares the same single Supabase project**. That's why every object here is `jobsitev2_`-prefixed: tables, functions, **and migrations must be globally unique across all sibling demos**, never just unique within this repo. A bare `record_spend` or `migrations/0001_*` name would collide with a sibling's. Rules: (1) **every** table/function/RPC/trigger is `jobsitev2_`-prefixed; (2) migration SQL only ever `CREATE`s/alters `jobsitev2_*` objects and is idempotent (`IF NOT EXISTS`); (3) **never** rename/drop a `jobsitev2_*` object expecting it to be the only thing in the project, and **never** touch un-prefixed or other apps' objects ‚Äî in particular the older `jobenrich_*` tables (from the boilerplate's own deployment) are left intact, NOT renamed. The **free-tier quotas are shared** across all demos (DB size, the 7-day inactivity pause, connection pool) ‚Äî keep usage light and prefer the REST transport (already used) over raw connections.

The app **cannot reach Orthogonal** until a real key is dropped in. The waterfall pipeline was validated out-of-band through the Orthogonal MCP tools (job extract, ContactOut search, and the Apollo/Bytemine/ContactOut enrich steps each confirmed to return live data). ## Architecture / data flow

**This is a public demo ‚Äî no login.** Access is controlled by Level 1 abuse protections (audit-driven), all in `lib/security/`:
- `guard.ts` ‚Äî `guardRequest(request, body, step)` runs at the top of every API route before any Orthogonal call: same-origin + content-type check, obvious-bot UA filter, CSRF-style request-token verify, honeypot, minimum form-timing (form steps), global daily **spend cap** (cheap early read ‚Üí 503), then per-visitor **per-step rate limit** (429). On success it returns **`reserveSpend()`** ‚Äî the route calls it after input validation and before the upstream call; it **atomically** reserves the step's cost against the cap and returns a 503 to send back if that tips over the cap (else null). (Replaces the old non-atomic `recordSpend()`.)
- `rateLimit.ts` / `spendCap.ts` ‚Äî **Supabase-backed when configured, in-memory otherwise.** Both check `getSupabase()` (`supabase.ts`): if `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` are set, the `$40` cap and the per-visitor `10/step/day` limit go through atomic Postgres RPCs (**`jobsitev2_try_reserve_spend`** ‚Äî one locked check-and-increment, so concurrent requests can't race past the cap ‚Äî and `jobsitev2_check_rate_limit`, in `supabase/migrations/0001_jobsitev2_*.sql`) so they hold across **all** serverless instances; unset ‚Üí module-level Map/counter (per-instance, leaks to `$40 √ó live instances`, zeroed on redeploy ‚Äî fine for a low-traffic demo). **On any Supabase error both fail *degraded* to the in-memory path**, so a paused/unreachable DB never takes the site down ‚Äî worst case the cap reverts to its old soft behavior. `guardRequest` is therefore `async`; routes `await` it and `await guard.reserveSpend()`.
- `tokens.ts` + `app/api/init/route.ts` ‚Äî issue/verify the signed request token (CSRF) and page-load stamp (timing); the client fetches them on mount.
- `client.ts` ‚Äî client helper: builds the composite fingerprint (IP + localStorage session token + UA/screen/tz/lang/platform), primes the token, and `apiPost()`s with all signals. `errorMessage()` renders 429/503 messages.
- Limits: **10 / step / day** per visitor, 24h reset; cap **$40/day**.

**Everything runs on a waterfall model** (ported from a sister demo): within each finder, a step only fires if the previous step returned **zero** people. Cheap/broad sources are tried first, and a fallback only costs money when it's actually needed. All finders live in `lib/people.ts` and return a normalized `Person` ({ name, title, linkedinUrl, profilePictureUrl, source }).

**Search flow** (`app/page.tsx` ‚Üí `app/api/search/route.ts`) ‚Äî input is **any job/careers URL** (LinkedIn, Indeed, Greenhouse, Lever, Workday, BambooHR, Gem, a company careers page, ‚Ä¶), gated only by `isValidJobUrl` (http(s) + host):
1. **`resolveJob(rawUrl)`** (`lib/jobResolver.ts`) turns any URL into `{ jobTitle, companyName, domain, jobLocation }` via a source waterfall ‚Äî cheap/structured first, LLM only when needed:
   - **LinkedIn** (`isLinkedInHost`) ‚Üí `canonicalizeLinkedInJobUrl` ‚Üí Edges `linkedin-extract-job` ($0.09; LinkedIn is auth-walled so the purpose-built extractor wins).
   - **Everything else** ‚Üí **Serper Scrape ($0.02)** which *renders JS* (so it works on SPAs like Workday/BambooHR/Gem that return an empty shell to a plain fetch) ‚Üí (a) parse a schema.org `JobPosting` **JSON-LD** if present (free, most reliable), else (b) LLM-extract from the rendered markdown via **ScrapeGraphAI `/api/extract` ($0.025)**.
   - `normalizeCompany()` strips legal suffixes ("Crocs, Inc." ‚Üí "Crocs") so the people providers match; `ATS_HOSTS` guard prevents using `bamboohr.com`/`myworkdayjobs.com`/etc. as the company's own domain. `jobTitle` may be **null** (a careers index with no single job) ‚Üí the people search runs **company-only**. `jobLocation` is extracted from all three paths (LinkedIn payload fields, JSON-LD `jobLocation.address`, LLM schema) and passed to the recruiter finder to avoid surfacing international recruiters for a US role. Throws on hard upstream failure (502); empty company ‚Üí 422.
2. `simplifyJobTitle()` strips intern/seasonal decorations from the resolved title so ContactOut can match.
3. Run two waterfalls in parallel (`Promise.allSettled`), each with its own error flag. **A resolved `domain` is an unambiguous company match; the `company` name is not** (e.g. "Orthogonal" ‚Üí the YC W26 company *and* the unrelated orthogonal.io). So the finders are **domain-first**: when a domain exists they try it FIRST, then fall back to the company name **only if the domain found nobody**. The same-namesake risk only applies when the domain returns the *wrong* people ‚Äî when it returns *none* (e.g. a slightly-off domain a provider doesn't index), a name match beats an empty result. The domain is PSL-reduced to its apex (`careers.sharkninja.com` ‚Üí `sharkninja.com`) in `lib/jobResolver.ts` so it's a domain the people providers actually index ‚Äî getting this wrong was the cause of "domain resolved but 0 people surfaced".
   - **People (target 5):** *domain present* ‚Üí ContactOut `people/search` (domain + title) ‚Üí (domain only) ‚Üí then the name fallback below. *No domain (or domain found nobody)* ‚Üí ContactOut (company + title) ‚Üí (company only) ‚Üí Coresignal `employee_base/search/filter/preview` (company).
   - **Recruiters (target 3):** Each ContactOut step is tried **with the job's location filter first** (e.g. "Boston, MA, United States") ‚Äî if that returns nobody the waterfall falls through to the same step without the filter, so a too-specific location never blocks results entirely. Virtual locations ("Remote", "Worldwide", etc.) are skipped. *domain present* ‚Üí ContactOut (domain + 13 recruiter titles [+ location]) ‚Üí (domain + titles, no location) ‚Üí then the name fallback below. *No domain (or domain found nobody)* ‚Üí ContactOut (company + recruiter titles [+ location]) ‚Üí (company + titles, no location) ‚Üí Coresignal (company + title "Recruiter") ‚Üí Coresignal (company only ‚Äî founders hire directly).

**Alumni flow** (`components/AlumniFinder.tsx` ‚Üí `app/api/alumni/route.ts`) ‚Äî opt-in secondary action; the school is **not** asked for up front. Takes company + domain (from the search response) + school. Same domain-first rule: *domain present* ‚Üí ContactOut (domain + education) ‚Üí then the company-name fallback if that found nobody; *no domain* ‚Üí ContactOut (company + education).

**Enrich flow** (`PersonCard` "Get contact" ‚Üí `app/api/enrich/route.ts` ‚Üí `EnrichDrawer`) ‚Äî cheap‚Üírich‚Üílast-resort contact waterfall, each step firing only if no email yet: **Apollo `/api/v1/people/match` ($0.01)** for a verified email + profile, straight from the LinkedIn URL ‚Üí **Bytemine `/contacts/enrich` ($0.03)** for work+personal email **and** mobile+work phone ‚Üí **ContactOut `/v1/people/linkedin` ($0.33, `include_phone:false`)** as the last resort. Returns `{ emails, phones, source, company, position, location, links }`, where `source: "apollo" | "bytemine" | "contactout" | "none"`. Each step is independently `try/catch`'d ‚Äî a provider error degrades to the next instead of failing the request ‚Äî and profile context, phones, and links **accumulate** across steps, so an email found late still carries context an earlier step surfaced. **Phones come from Bytemine (mobile+work) and ContactOut, not Apollo** ‚Äî Apollo's phone reveal is intentionally not requested so it stays a flat $0.01; ContactOut's phone tier ($0.55) is likewise unused (`include_phone` stays false). Apollo's raw payload is ~190 KB, so the route extracts only the fields it needs via `apolloLookup` and **never logs the whole body**; `isRealEmail()` drops Apollo's `email_not_unlocked@domain.com` placeholder. Provider response shapes are inconsistent, so the route coerces defensively (`collectStrings`, `cleanStr`, `asUrl`, `joinLocation`) ‚Äî preserve that. The UI never names the providers (no "Apollo/Bytemine/ContactOut" in progress or results). Results are cached client-side per LinkedIn URL in `page.tsx` (`enrichCache`): the first "Get contact" fetches; afterward the button reads "Open contact" and reopening the drawer is instant and free.

**Coresignal caveat:** `experience_company_name` matches anyone who *ever* worked there, so `fromCoresignal()` filters results to rows whose *current* `company_name` matches the target (falling back to the raw list only if none match). Coresignal returns `profile_url` (a real LinkedIn URL), so its results are still enrichable.

**v2 ‚Äî role-first job search flow** (`app/jobs/page.tsx` ‚Üí `app/api/jobs/search/route.ts` ‚Üí `lib/jobs/searchJobs.ts`) ‚Äî the alternative entry point: instead of pasting a URL, the user types a **role** (+ optional location / employment type / remote / posted-within) and gets a list of live postings, each with a "Find people" button that runs the v1 people flow.
1. **`searchJobs(params)`** calls Fantastic Jobs (api `fantastic-jobs`) via the same `callOrthogonal` wrapper ‚Äî one GET, **$0.40 flat regardless of result count** (so use a generous `limit`; default 25). Response is a **bare array** at `json.data`. **TWO interchangeable feeds, same shape, chosen by `feedPath(params)`:** `/v1/active-ats` (default ‚Äî 54 company career sites: Greenhouse/Lever/Ashby/Workday/Gem‚Ä¶; best for full-time, thin on interns) and `/v1/active-jb` (LinkedIn + Wellfound + Y Combinator; far better intern/entry/startup coverage ‚Äî verified live). **Intern queries (`employmentType==="INTERN"`) auto-route to `active-jb`**; an explicit `params.board` (`"ats"`/`"jb"`) overrides. The only shape difference: `active-jb` has **no `domain_derived`** ‚Äî the employer domain is `org_linkedin_website` there, so `normalizeJob` reads `domain_derived ?? org_linkedin_website`. `normalizeJob()` maps the confirmed real keys (`organization`‚Üícompany, the domain field above [apex; ATS hosts‚Üínull], `locations_derived`, `ai_work_arrangement`, `ai_employment_type`, `ai_experience_level`, `ai_salary_*`, `organization_logo`) into `JobListing` (`types/job.ts`). Param map: `role‚Üítitle`, `location‚Üílocation`, `remoteOnly‚Üíai_work_arrangement="Remote OK"`, `employmentType‚Üíai_employment_type`, `freshness‚Üítime_frame` (default `7d`); pagination via `cursor`. The route is **POST** (not GET as first sketched) so the guard can read fp/honeypot/token from the JSON body; it runs through `guardRequest(‚Ä¶, "jobsSearch")` ($0.40 against the shared cap).
   - **Empty results are usually a too-narrow filter combo, not a failure** (e.g. "Product Manager" + Internship + San Francisco is genuinely ~0 across providers ‚Äî confirmed live; SWE-intern in the same city returns plenty). The page's empty state names the exact query and lists which filters to loosen, plus a one-tap **"Search LinkedIn & startups instead"** button that re-runs on `active-jb` ‚Äî a deliberate click, so no automatic second $0.40 call.
   - **Why not Coresignal as a cheaper source:** Coresignal job *search* is $0.021 but returns no URLs; getting a clickable posting needs Collect at **$0.042/job**, so a 25-listing page ‚âà $1.07 vs Fantastic Jobs' $0.40-flat-for-up-to-1000. Coresignal is cheaper per *search* but ~2.7√ó more expensive per *listing page* ‚Äî keep Fantastic Jobs. (`created_at_gte` there wants `YYYY-MM-DD HH:MM:SS`, not the date-only the docs claim.)
2. **"Find people"** reuses `app/api/search/route.ts` via its **preset path**: the route accepts optional `companyName`/`companyDomain`/`jobTitle`/`jobLocation` and, when `companyName` is present, **skips `resolveJob`** (no scrape/LLM ‚Äî Fantastic Jobs already gave us structured company data) and runs the people/recruiter waterfalls directly. The client MUST call `/api/search` with `apiPost(..., { timed: true })` ‚Äî the `search` step is timing-gated, so omitting `pageLoad` 403s. (Chose Option B over re-resolving the posting URL: saves a redundant resolve call per click.)
3. **Session model:** see the `sessionStorage` / in-app-tabs note under Scaling ‚Üí Caching. The page keeps multiple search "tabs" (each = params + listings + per-listing people), a **session-wide** `searchCache` (re-running an identical search is free) and `enrichCache` (shared across tabs), all persisted to `sessionStorage`. **No server-side cache.**

- **People + Recruiters** (each ContactOut step $0.05, Coresignal $0.021; run in parallel; best $0.05 when the first step hits): *no domain* ‚Üí People ‚â§ $0.121 (company+title ‚Üí company ‚Üí Coresignal), Recruiters ‚â§ $0.142 (company+titles+location ‚Üí company+titles ‚Üí Coresignal recruiter ‚Üí Coresignal all). *domain present* ‚Üí domain steps run first, name fallback only if nobody found: People ‚â§ $0.221, Recruiters ‚â§ $0.242 worst case (location-filtered step + unfiltered step at each level). The location-filtered step is an extra $0.05 only when it returns nobody; the common path (location-filtered first step hits) is still $0.05.
- **Search total:** LinkedIn **$0.19** best / ~$0.45 worst; careers/Greenhouse **~$0.12** best / ~$0.41 worst (worst = resolve + both finders exhausting their domain *and* name fallbacks; only when nobody is found at any step).

- **v2 role-first search (`/jobs`):** the search step is **Fantastic Jobs `/v1/active-ats` = $0.40 flat per search** (one call; price does NOT scale with result count ‚Äî `limit:5` and `limit:1000` both cost $0.40, so use a generous page size). "Find people" on a listing reuses the v1 people/recruiter waterfall via the **preset path** (skips URL-resolve since company+domain are already known), so it's *cheaper* than v1: people+recruiters ~$0.05‚Äì0.12, then enrich per contact $0.01‚Äì0.37. ## Scaling / deployment (audited 2026-06-16)

Target: Vercel project **jobsite-v2**, **Hobby plan**, with the shared **Supabase free** project for the cap/rate-limiter. The new Vercel project must have `ORTHOGONAL_API_KEY` (`orth_live_‚Ä¶`), `REQUEST_TOKEN_SECRET`, `NEXT_PUBLIC_APP_URL`, and the `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` pair set, and the `0001_jobsitev2_*` migration run, before it can serve traffic. Verdict: **fine for ~100 users**; for a public scale launch see the Hobby ToS note below.

- **~100 users: nothing breaks.** Spend ‚âà $20‚Äì30 total (under the cap). Gates 1‚Äì6 (origin/token/honeypot/timing/bot) are airtight regardless of instance count.
- **Global hard cap ‚Äî implemented (Level 2), now atomic.** `spendCap.ts`/`rateLimit.ts` use Supabase via `@supabase/supabase-js` REST (the `jobsitev2_try_reserve_spend` / `jobsitev2_check_rate_limit` RPCs on `jobsitev2_`-prefixed tables, so one Supabase project can be shared across demos without colliding) when `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` are set, so the `$40` cap and `10/step/day` limit hold across all instances; otherwise they use the in-memory counters. The cap is reserved by a **single locked check-and-increment** (`reserveSpend`), closing the read-then-write TOCTOU that previously let concurrent requests overshoot. **Both fail *degraded* to in-memory on any Supabase error** (so a paused/unreachable DB never 500s the site). Supabase free pauses after 7 days idle (~30s resume) ‚Äî an active demo won't pause; add a keep-alive cron if it ever goes quiet. **Setup:** run `supabase/migrations/0001_jobsitev2_spend_and_rate_limit.sql` in the Supabase SQL editor, then set the two env vars in Vercel.
- **Vercel Hobby is non-commercial-only** ([ToS](https://vercel.com/legal/terms)). A public, Orthogonal-branded demo built by an Orthogonal employee is commercial; Vercel may shut it down without notice, and Hobby pauses (no overage billing) if a spike exceeds included usage. **A public scale launch needs Vercel Pro ($20/mo).** (Hobby+Fluid-compute allows `maxDuration` up to 300s, so the `maxDuration = 60` on `search`/`alumni` is fine.)
- **Caching is intentionally per-user/session only.** `enrichCache` (client-side per browser) makes a re-opened contact free *for that user*; there is deliberately **no cross-user cache** ‚Äî a scrape/search/enrich is **never** saved and re-served to a different visitor (privacy + product decision). This means overlapping searches across users each pay full price; that's accepted. `$40/day` therefore covers ~130‚Äì200 full sessions/day.
  - **v2 job search (`/jobs`) session model ‚Äî `sessionStorage`, one browser tab = one session.** The role-first page keeps multiple *in-app* search "tabs" (each = one search's params + listings + per-listing people results) so a user can run several searches and switch back to an earlier one with its listings intact. All of it (search results, people results, and the **session-wide** `enrichCache` shared across in-app tabs) is persisted to **`sessionStorage`**: it survives reloads/navigation *within* that browser tab but a new browser tab is a fresh empty session and closing the tab wipes it. This is the policy-compliant choice ‚Äî `sessionStorage` is per-tab and never shared across users, so it honors the no-cross-user-persistence rule with **no Supabase cache**. (The Fantastic Jobs endpoint refreshes hourly upstream, so a per-session cache loses no meaningful freshness.) **Do not** add a server-side / cross-user `job_search_cache` table ‚Äî that would violate the data policy above.

## Key files

- `lib/orthogonal.ts` ‚Äî the single `callOrthogonal()` wrapper. **All external API calls go through this.** Hits `api.orthogonal.com/v1/run` with `{ api, path, method, body?, query? }`, throws an **`OrthogonalError`** on `!res.ok` or `!json.success`, returns `json.data` (the provider's raw response body). `OrthogonalError.quotaExceeded` is set on a 402 / payment / quota signal; `isQuotaError(err)` + `QUOTA_MSG` let routes surface a friendly 503 instead of a generic 502.
- `lib/domains.ts` ‚Äî **shared** host/company helpers used by both v1 and v2: `ATS_HOSTS` / `NON_COMPANY_HOSTS` / `isInfraHost`, `hostFromUrl` (PSL-reduce any host to its registrable apex via `tldts` ‚Äî `careers.x.com`/`jobs.x.co.uk` ‚Üí `x.com`/`x.co.uk`; don't hand-roll subdomain stripping), `normalizeCompanyDomain`, `pickDomain`, and `normalizeCompany` (strip legal suffixes). This is the single source of truth ‚Äî `jobResolver.ts` and `jobs/searchJobs.ts` import from here (the old duplicated copies are gone).

## Conventions

- Validate input and return `NextResponse.json({ error }, { status })` on bad requests; use 502 for upstream Orthogonal failures, and 503 (`QUOTA_MSG`, via `isQuotaError`) when the failure is the API key's own spend/quota limit.
- Baseline security headers (clickjacking / nosniff / referrer / permissions / HSTS) are set for all routes in `next.config.ts`. No CSP (the App Router injects un-nonced inline styles/scripts). CI (`.github/workflows/ci.yml`) runs `tsc --noEmit` + `npm run build` on every PR.
- **Error text surfacing:** routes put user-facing text in `error`; the security guard puts it in `message`. The client's `errorMessage()` (`lib/security/client.ts`) reads **both** ‚Äî keep it that way, or route errors silently fall back to the generic UI message. A URL that resolves to no company (e.g. application portals like ADP/iCIMS/Taleo that block scraping) returns 422 with `UNREADABLE_MSG`, which tells the user to try a LinkedIn/Greenhouse/Workday/careers link instead.
- API routes log with a `[search]` / `[jobs]` / `[alumni]` / `[enrich]` / `[people]` prefix.
- New external data sources go through `callOrthogonal`, not raw `fetch`, and new people-finders go through `lib/people.ts`'s `waterfall()` so the fire-only-on-empty cost discipline is preserved.
