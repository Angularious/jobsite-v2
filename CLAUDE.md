# jobsite-v2

> **Repo:** `Angularious/jobsite-v2` — the canonical repo. `jobenrich` was the
> boilerplate this was copied from; ignore it. **Deploy target:** Vercel project
> **jobsite-v2**. **Supabase objects:** `jobsitev2_`-prefixed. The user-facing UI
> still reads "Job Enrich" (`/`) / "Job Search" (`/jobs`) — renaming that public
> copy is a separate product decision, intentionally not done as part of the

A Next.js 16 app: a **role-first** job search. A student/job-seeker types a **role** (+ optional location / type / remote / recency) and gets live postings from LinkedIn, Wellfound, YC, and 54 ATS platforms; on any listing they hit **Find people** to surface people in similar roles + recruiters at that company, and pull contact info for anyone on demand. **Powered by Orthogonal.** (The older URL-paste flow lives separately at jobenrich.vercel.app.)

## Stack

- Next.js 16.2.9 (App Router) · React 19 · TypeScript
- Tailwind v4 — config lives in `app/globals.css` via `@theme`, **not** a `tailwind.config` file. Design system is **light raw brutalism** (per the `nextlevelbuilder/ui-ux-pro-max-skill` "Brutalism" spec): white paper, black ink + 3px black borders, blue underlined links, pure primary accents (red is primary; blue/yellow/green/pink alongside), **no shadows**, zero border-radius, system-ui + monospace fonts (bold 700+), instant state changes (`transition: none`). Buttons are plain bordered blocks that **invert on hover** (white/black → black/white) — colors live in the `.nb-btn` primitive, so they override utility bg/text classes on the element. Reusable primitives `.nb-card` / `.nb-btn` / `.nb-input` / `.nb-flat` live in `globals.css`; a fill accent (where needed) is passed via the inline `--nb` CSS custom property. Errors/alerts use hot pink to stay distinct from the red primary. **Gotcha:** `text-base` resolves to the `--color-base` **color** (white), not a font-size — use `text-ink` for black text and an explicit size class (e.g. `text-sm`).
- `lucide-react` for icons. No test framework.

## Environment variables

- `ORTHOGONAL_API_KEY` — auth for all external data calls (required, **server-side only**, never `NEXT_PUBLIC_`).
- `DAILY_SPEND_CAP_USD` — hard daily ceiling across all visitors (default 40 in code). Editable in Vercel without a code change.
- `REQUEST_TOKEN_SECRET` — HMAC secret for signing request/timing tokens (random 32+ chars).
- `NEXT_PUBLIC_APP_URL` — public deployment origin, used for same-origin checks on API routes.
- `SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` — **optional**. When both are set (and `supabase/migrations/0001_*.sql` has been run), the spend cap + rate limiter use Supabase so they hold across all serverless instances. Unset → the in-memory per-instance counters are used (fine for a low-traffic demo). Service-role key is server-side only — never `NEXT_PUBLIC_`. Free tier is sufficient.

> **⚠ Shared Supabase project — ONE Supabase project for ALL demo sites in `~/Projects/`:** every Orthogonal demo site in the `Projects/` folder (one level above `jobsearch/`) — `rolodex`, `job-intel`, `jobenrich`, etc. — **shares the same single Supabase project**. That's why every object here is `jobsitev2_`-prefixed: tables, functions, **and migrations must be globally unique across all sibling demos**, never just unique within this repo. A bare `record_spend` or `migrations/0001_*` name would collide with a sibling's. Rules: (1) **every** table/function/RPC/trigger is `jobsitev2_`-prefixed; (2) migration SQL only ever `CREATE`s/alters `jobsitev2_*` objects and is idempotent (`IF NOT EXISTS`); (3) **never** rename/drop a `jobsitev2_*` object expecting it to be the only thing in the project, and **never** touch un-prefixed or other apps' objects — in particular the older `jobenrich_*` tables (from the boilerplate's own deployment) are left intact, NOT renamed. The **free-tier quotas are shared** across all demos (DB size, the 7-day inactivity pause, connection pool) — keep usage light and prefer the REST transport (already used) over raw connections.

## Architecture / data flow

**This is a public demo — no login.** Access is controlled by Level 1 abuse protections (audit-driven), all in `lib/security/`:
- `guard.ts` — `guardRequest(request, body, step)` runs at the top of every API route before any Orthogonal call: same-origin + content-type check, obvious-bot UA filter, CSRF-style request-token verify, honeypot, minimum form-timing (form steps), global daily **spend cap** (cheap early read → 503), then per-visitor **per-step rate limit** (429). On success it returns **`reserveSpend()`** — the route calls it after input validation and before the upstream call; it **atomically** reserves the step's cost against the cap and returns a 503 to send back if that tips over the cap (else null). (Replaces the old non-atomic `recordSpend()`.)
- `rateLimit.ts` / `spendCap.ts` — **Supabase-backed when configured, in-memory otherwise.** Both check `getSupabase()` (`supabase.ts`): if `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` are set, the `$40` cap and the per-visitor `10/step/day` limit go through atomic Postgres RPCs (**`jobsitev2_try_reserve_spend`** — one locked check-and-increment, so concurrent requests can't race past the cap — and `jobsitev2_check_rate_limit`, in `supabase/migrations/0001_jobsitev2_*.sql`) so they hold across **all** serverless instances; unset → module-level Map/counter (per-instance, leaks to `$40 × live instances`, zeroed on redeploy — fine for a low-traffic demo). **On any Supabase error both fail *degraded* to the in-memory path**, so a paused/unreachable DB never takes the site down — worst case the cap reverts to its old soft behavior. `guardRequest` is therefore `async`; routes `await` it and `await guard.reserveSpend()`.
- `tokens.ts` + `app/api/init/route.ts` — issue/verify the signed request token (CSRF) and page-load stamp (timing); the client fetches them on mount.
- `client.ts` — client helper: builds the composite fingerprint (IP + localStorage session token + UA/screen/tz/lang/platform), primes the token, and `apiPost()`s with all signals. `errorMessage()` renders 429/503 messages.
- Limits: **10 / step / day** per visitor, 24h reset; cap **$40/day**.

**Everything runs on a waterfall model** (ported from a sister demo): within each finder, a step only fires if the previous step returned **zero** people. Cheap/broad sources are tried first, and a fallback only costs money when it's actually needed. All finders live in `lib/people.ts` and return a normalized `Person` ({ name, title, linkedinUrl, profilePictureUrl, source }).

> **The URL-paste flow (v1) and the alumni finder have been REMOVED from this repo.** Pasting a job URL (the old `resolveJob` scrape/LLM pipeline, `lib/jobResolver.ts`, `/api/alumni`, `SearchForm`/`SampleResults`/`AlumniFinder`) now lives on a separate deployment, **jobenrich.vercel.app**, linked from the header as "Search by URL". This repo is the **role-first** product only. The people/recruiter waterfall and the enrich flow below are shared and stay.

**Search flow (role-first — the home page)** — see the "role-first job search flow" section below; it is now the primary and only in-app flow. `/api/search` is **preset-only** (`companyName` is required; it no longer resolves URLs) and runs the people + recruiter waterfalls:
- **People (target 5):** *domain present* → ContactOut `people/search` (domain + title) → (domain only) → then the name fallback. *No domain (or domain found nobody)* → ContactOut (company + title) → (company only) → Coresignal `employee_base/search/filter/preview` (company).
- **Recruiters (target 3):** each ContactOut step is tried **with the job's location filter first**; if that returns nobody the waterfall falls through to the same step without the filter. Virtual locations ("Remote"/"Worldwide") are skipped. *domain present* → ContactOut (domain + recruiter titles [+ location]) → (domain + titles, no location) → name fallback. *No domain* → ContactOut (company + recruiter titles [+ location]) → (company + titles) → Coresignal (company + "Recruiter") → Coresignal (company only — founders hire directly).
- **Domain-first rationale:** a resolved `domain` is an unambiguous company match; the `company` name is not (namesakes). So finders try the domain FIRST and fall back to the name only if the domain found nobody. The domain comes pre-apex-reduced from Fantastic Jobs.

**Enrich flow** (`PersonCard` "Get contact" → `app/api/enrich/route.ts` → `EnrichDrawer`) — cheap→rich→last-resort contact waterfall, each step firing only if no email yet: **Apollo `/api/v1/people/match` ($0.01)** for a verified email + profile, straight from the LinkedIn URL → **Bytemine `/contacts/enrich` ($0.03)** for work+personal email **and** mobile+work phone → **ContactOut `/v1/people/linkedin` ($0.33, `include_phone:false`)** as the last resort. Returns `{ emails, phones, source, company, position, location, links }`, where `source: "apollo" | "bytemine" | "contactout" | "none"`. Each step is independently `try/catch`'d — a provider error degrades to the next instead of failing the request — and profile context, phones, and links **accumulate** across steps, so an email found late still carries context an earlier step surfaced. **Phones come from Bytemine (mobile+work) and ContactOut, not Apollo** — Apollo's phone reveal is intentionally not requested so it stays a flat $0.01; ContactOut's phone tier ($0.55) is likewise unused (`include_phone` stays false). Apollo's raw payload is ~190 KB, so the route extracts only the fields it needs via `apolloLookup` and **never logs the whole body**; `isRealEmail()` drops Apollo's `email_not_unlocked@domain.com` placeholder. Provider response shapes are inconsistent, so the route coerces defensively (`collectStrings`, `cleanStr`, `asUrl`, `joinLocation`) — preserve that. The UI never names the providers (no "Apollo/Bytemine/ContactOut" in progress or results). Results are cached client-side per LinkedIn URL in `page.tsx` (`enrichCache`): the first "Get contact" fetches; afterward the button reads "Open contact" and reopening the drawer is instant and free.

**Coresignal caveat:** `experience_company_name` matches anyone who *ever* worked there, so `fromCoresignal()` filters results to rows whose *current* `company_name` matches the target (falling back to the raw list only if none match). Coresignal returns `profile_url` (a real LinkedIn URL), so its results are still enrichable.

**Role-first job search flow** (`app/page.tsx` [the home page] → `app/api/jobs/search/route.ts` → `lib/jobs/searchJobs.ts`) — the user types a **role** (+ optional location / employment type / remote / posted-within) and gets a list of live postings, each with a "Find people" button that runs the people/recruiter flow.
1. **`searchJobs(params)`** calls Fantastic Jobs (api `fantastic-jobs`) via the same `callOrthogonal` wrapper — one GET, **$0.40 flat regardless of result count** (so use a generous `limit`; default 25). Response is a **bare array** at `json.data`. **TWO interchangeable feeds, same shape, chosen by `feedPath(params)`:** `/v1/active-ats` (default — 54 company career sites: Greenhouse/Lever/Ashby/Workday/Gem…; best for full-time, thin on interns) and `/v1/active-jb` (LinkedIn + Wellfound + Y Combinator; far better intern/entry/startup coverage — verified live). **Intern queries (`employmentType==="INTERN"`) auto-route to `active-jb`**; an explicit `params.board` (`"ats"`/`"jb"`) overrides. `active-jb` has **no `domain_derived`** — the employer domain is `org_linkedin_website` there, so `normalizeJob` reads `domain_derived ?? org_linkedin_website`. Param map: `role→title` (**via `expandRole()`** — see below), `location→location`, `remoteOnly→ai_work_arrangement="Remote OK"`, `employmentType→ai_employment_type`, `freshness→time_frame` (default `7d`); pagination via `cursor`. The route is **POST** so the guard can read fp/honeypot/token from the JSON body; it runs through `guardRequest(…, "jobsSearch")` ($0.40 against the shared cap).
   - **Abbreviation expansion (`expandRole`, free):** Fantastic Jobs matches `title` fairly literally, so "GTM" matched ~no posting titles (they read "Go-to-Market Intern"/"Growth Marketing Intern"). `expandRole` rewrites a curated set of whole-word abbreviations (GTM→"go to market", SWE→"software engineer", PM→"product manager", …) before querying. **Diagnosed live:** `title="GTM"`+SF+intern+24h → 0; `title="marketing"`+SF+intern+7d → 7 real GTM-adjacent intern roles. So empties were query-semantics + a tight window, not a Fantastic Jobs coverage gap.
   - **Serper web fallback (`lib/jobs/webSearch.ts`, $0.002):** when the structured feed returns **0** listings (and it's a first page, not pagination), the route fires a Serper Google Search and returns raw web links (`WebResult[]`) — LinkedIn/Indeed/Greenhouse/aggregators — so the user still has somewhere to go. These are **not** enriched listings (no company/apply/date), rendered as a plain link list in the empty state. Fantastic Jobs itself stays exactly one $0.40 call; Serper is the only extra (negligible).
   - **Empty results are usually a too-narrow filter combo, not a failure.** The empty state names the exact query, lists which filters to loosen, shows the Serper web links, and offers a one-tap **"Search LinkedIn & startups instead"** button (re-runs on `active-jb`) — a deliberate click, so no automatic second $0.40 call.
   - **Why not Coresignal as a cheaper source:** Coresignal job *search* is $0.021 but returns no URLs; a clickable posting needs Collect at **$0.042/job** (~$1.07 for 25 vs Fantastic Jobs' $0.40-flat) — keep Fantastic Jobs.
2. **"Find people"** calls `app/api/search/route.ts` (preset-only): it takes `companyName`/`companyDomain`/`jobTitle`/`jobLocation` from the listing and runs the people/recruiter waterfalls directly (no URL resolve). The client MUST call it with `apiPost(..., { timed: true })` — the `search` step is timing-gated, so omitting `pageLoad` 403s.
3. **Session model:** see the `sessionStorage` / in-app-tabs note under Scaling → Caching. The page keeps multiple search "tabs" (each = params + listings + per-listing people), a **session-wide** `searchCache` (re-running an identical search is free) and `enrichCache` (shared across tabs), all persisted to `sessionStorage`. **No server-side cache.**

## Costs (best case = first step hits, the common path; all unit prices verified live against the Orthogonal marketplace)

- **Resolve (varies by source):** LinkedIn = **$0.09** (Edges); everything else = **$0.02** (Serper, JSON-LD hit) → **$0.045** (Serper + ScrapeGraphAI LLM fallback). Non-LinkedIn is *cheaper* than LinkedIn.
- **People + Recruiters** (each ContactOut step $0.05, Coresignal $0.021; run in parallel; best $0.05 when the first step hits): *no domain* → People ≤ $0.121 (company+title → company → Coresignal), Recruiters ≤ $0.142 (company+titles+location → company+titles → Coresignal recruiter → Coresignal all). *domain present* → domain steps run first, name fallback only if nobody found: People ≤ $0.221, Recruiters ≤ $0.242 worst case (location-filtered step + unfiltered step at each level). The location-filtered step is an extra $0.05 only when it returns nobody; the common path (location-filtered first step hits) is still $0.05.
- **Search total:** LinkedIn **$0.19** best / ~$0.45 worst; careers/Greenhouse **~$0.12** best / ~$0.41 worst (worst = resolve + both finders exhausting their domain *and* name fallbacks; only when nobody is found at any step).
- **Alumni:** $0.05 (→ $0.10 worst case with the domain fallback).
- **Enrich (per contact):** $0.01 (Apollo hit) → $0.04 (Apollo miss + Bytemine) → $0.37 worst case (both miss + ContactOut $0.33). Cached client-side, so paid once per profile.
- **Enrich all returned (≤5 people + ≤3 recruiters = 8):** ~$0.08 best (all Apollo) → ~$0.32 mid → ~$2.96 worst (all ContactOut).
- **Full session (paste + enrich everyone):** ~$0.20–0.27 typical → ~$3.2 absolute worst. The $40/day cap covers ~130–200 typical *v1 (URL-paste)* sessions/day (soft, in-memory).
- **v2 role-first search (`/jobs`):** the search step is **Fantastic Jobs `/v1/active-ats` = $0.40 flat per search** (one call; price does NOT scale with result count — `limit:5` and `limit:1000` both cost $0.40, so use a generous page size). "Find people" on a listing reuses the v1 people/recruiter waterfall via the **preset path** (skips URL-resolve since company+domain are already known), so it's *cheaper* than v1: people+recruiters ~$0.05–0.12, then enrich per contact $0.01–0.37. A full v2 session ≈ **$0.40 (search) + $0.12 (people) + ~$0.10–0.40 (enriches) ≈ $0.60–0.90**. **`DAILY_SPEND_CAP_USD` kept at $40** → ~**45–65 full v2 sessions/day** before the cap trips (then a 503 "try tomorrow"); the cap is the credit backstop and Jerry chose to keep it at $40 for launch even though ~100 daily users doing full sessions could hit it. `sessionStorage` per-session caching means a user never pays twice for the same search/contact within their visit, which softens this.

> Dynamic-priced endpoints: ContactOut `/v1/people/search` is `reveal_info ? 25*0.75 : 0.05` and `/v1/people/linkedin` is `include_phone ? 0.55 : 0.33`; ScrapeGraphAI `/api/extract` adds +$0.025 with `stealth`. The app pins these to the cheap side.

## Scaling / deployment (live 2026-06-23)

**Live at https://jobsitekiller.vercel.app** — Vercel project **jobsite-v2** (`angularious` team), with the shared **Supabase free** project for the cap/rate-limiter. `ORTHOGONAL_API_KEY` (`orth_live_…`) + `REQUEST_TOKEN_SECRET` are set in Vercel prod; for the cross-instance cap, the `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` pair must be set and the `0001_jobsitev2_*` migration run (verify via the `jobsitev2_daily_spend` smoke test). Verdict: **fine for ~100 users**; for a public scale launch see the Hobby ToS note below.

- **~100 users: nothing breaks.** Spend ≈ $20–30 total (under the cap). Gates 1–6 (origin/token/honeypot/timing/bot) are airtight regardless of instance count.
- **Global hard cap — implemented (Level 2), now atomic.** `spendCap.ts`/`rateLimit.ts` use Supabase via `@supabase/supabase-js` REST (the `jobsitev2_try_reserve_spend` / `jobsitev2_check_rate_limit` RPCs on `jobsitev2_`-prefixed tables, so one Supabase project can be shared across demos without colliding) when `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` are set, so the `$40` cap and `10/step/day` limit hold across all instances; otherwise they use the in-memory counters. The cap is reserved by a **single locked check-and-increment** (`reserveSpend`), closing the read-then-write TOCTOU that previously let concurrent requests overshoot. **Both fail *degraded* to in-memory on any Supabase error** (so a paused/unreachable DB never 500s the site). Supabase free pauses after 7 days idle (~30s resume) — an active demo won't pause; add a keep-alive cron if it ever goes quiet. **Setup:** run `supabase/migrations/0001_jobsitev2_spend_and_rate_limit.sql` in the Supabase SQL editor, then set the two env vars in Vercel.
- **Vercel Hobby is non-commercial-only** ([ToS](https://vercel.com/legal/terms)). A public, Orthogonal-branded demo built by an Orthogonal employee is commercial; Vercel may shut it down without notice, and Hobby pauses (no overage billing) if a spike exceeds included usage. **A public scale launch needs Vercel Pro ($20/mo).** (Hobby+Fluid-compute allows `maxDuration` up to 300s, so the `maxDuration = 60` on `search` is fine.)
- **Caching is intentionally per-user/session only.** `enrichCache` (client-side per browser) makes a re-opened contact free *for that user*; there is deliberately **no cross-user cache** — a scrape/search/enrich is **never** saved and re-served to a different visitor (privacy + product decision). This means overlapping searches across users each pay full price; that's accepted. `$40/day` therefore covers ~130–200 full sessions/day.
  - **Job search session model — `sessionStorage`, one browser tab = one session.** The role-first home page (`app/page.tsx`) keeps multiple *in-app* search "tabs" (each = one search's params + listings + per-listing people results) so a user can run several searches and switch back to an earlier one with its listings intact. All of it (search results, people results, and the **session-wide** `enrichCache` shared across in-app tabs) is persisted to **`sessionStorage`**: it survives reloads/navigation *within* that browser tab but a new browser tab is a fresh empty session and closing the tab wipes it. This is the policy-compliant choice — `sessionStorage` is per-tab and never shared across users, so it honors the no-cross-user-persistence rule with **no Supabase cache**. (The Fantastic Jobs endpoint refreshes hourly upstream, so a per-session cache loses no meaningful freshness.) **Do not** add a server-side / cross-user `job_search_cache` table — that would violate the data policy above.

## Conventions

- Validate input and return `NextResponse.json({ error }, { status })` on bad requests; use 502 for upstream Orthogonal failures, and 503 (`QUOTA_MSG`, via `isQuotaError`) when the failure is the API key's own spend/quota limit.
- Baseline security headers (clickjacking / nosniff / referrer / permissions / HSTS) are set for all routes in `next.config.ts`. No CSP (the App Router injects un-nonced inline styles/scripts). CI (`.github/workflows/ci.yml`) runs `tsc --noEmit` + `npm run build` on every PR.
- **Error text surfacing:** routes put user-facing text in `error`; the security guard puts it in `message`. The client's `errorMessage()` (`lib/security/client.ts`) reads **both** — keep it that way, or route errors silently fall back to the generic UI message. A URL that resolves to no company (e.g. application portals like ADP/iCIMS/Taleo that block scraping) returns 422 with `UNREADABLE_MSG`, which tells the user to try a LinkedIn/Greenhouse/Workday/careers link instead.
- API routes log with a `[search]` / `[jobs]` / `[alumni]` / `[enrich]` / `[people]` prefix.
- New external data sources go through `callOrthogonal`, not raw `fetch`, and new people-finders go through `lib/people.ts`'s `waterfall()` so the fire-only-on-empty cost discipline is preserved.
